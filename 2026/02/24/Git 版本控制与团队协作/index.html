Git 版本控制与团队协作

Git 是现代软件开发中不可或缺的工具。本篇将介绍 Git 的核心概念、常用命令、分支管理和团队协作流程。

一、Git 基础

1.1 版本控制简介

版本控制系统（VCS） 用于跟踪文件的变化历史，主要类型：

类型 特点 代表
本地 VCS 单用户，单机器 RCS, SCCS
集中式 VCS 单服务器，多客户端 SVN, CVS
分布式 VCS 每人都有完整仓库 Git, Mercurial

1.2 Git 的特点

  • 分布式：每个人都是完整的仓库
  • 分支模型：轻量级分支，快速切换
  • 快照模式：存储快照而非差异
  • 数据校验：SHA-1 校验和保证数据完整性
  • 支持非线性开发：强大的合并能力

1.3 Git 的三个区域

  • 工作目录：你看到和编辑的文件
  • 暂存区：准备提交的快照
  • 仓库：永久存储的提交历史

5.3 Pull Request / Merge Request 流程

创建 PR 的步骤：

  1. Fork 仓库（如果是外部贡献）
  2. 克隆仓库：git clone your-fork
  3. 创建功能分支：git checkout -b feature-xxx
  4. 提交更改：git commit -m "描述"
  5. 推送分支：git push origin feature-xxx
  6. 在 GitHub/Gitee 上创建 PR
  7. 等待审查，根据意见修改
  8. PR 合并

PR 最佳实践：

  • 标题简洁明了
  • 描述说明更改内容和原因
  • 一次 PR 只做一件事
  • 确保 CI 测试通过
  • 及时回复审查意见

5.4 提交信息规范

Conventional Commits 规范：

Type 类型：

  • feat - 新功能
  • fix - 修复 bug
  • docs - 文档更新
  • style - 代码格式（不影响功能）
  • refactor - 重构
  • perf - 性能优化
  • test - 测试相关
  • chore - 构建/工具/配置
  • ci - CI 配置
  • revert - 回滚

七、最佳实践

提醒自己：

  1. 提交要频繁：小步提交，便于回滚和审查
  2. 提交信息要清晰：说明做了什么和为什么
  3. 分支命名要规范：feature/xxx, bugfix/xxx
  4. 及时同步主分支：避免大的合并冲突
  5. 代码审查要认真：及时回复，互相学习
  6. 不要推送修改的历史：避免给别人造成困扰
  7. 定期清理分支：删除已合并的分支
  8. 使用.gitignore：不提交不必要的文件
