MySQL 进阶教程 07-01 索引详解

索引是数据库性能优化的核心技术。本文将详细介绍索引的概念、数据结构、分类、创建和使用方法，以及如何使用 EXPLAIN 分析查询性能。

一、索引概述

1.1 什么是索引？

索引（Index） 是帮助数据库高效获取数据的数据结构。类似于书籍的目录，通过索引可以快速定位到目标数据，而无需扫描整个表。

1.2 索引的优缺点

优点：

  • 大幅提高查询速度（最主要目的）
  • 加速表连接（JOIN）操作
  • 加速 ORDER BY 和 GROUP BY
  • 实现唯一性约束

缺点：

  • 占用额外存储空间
  • 降低 INSERT、UPDATE、DELETE 速度（需要同时更新索引）
  • 创建和维护需要时间

1.3 索引使用场景

场景 说明
主键/外键列 自动创建索引
WHERE 条件列 经常作为查询条件的列
JOIN 连接列 表连接的关联列
ORDER BY 列 排序字段
GROUP BY 列 分组字段
范围查询 使用 BETWEEN、>、< 等

不适合创建索引的场景：

  • 数据量很小的表
  • 频繁更新的列
  • 选择性低的列（如性别、状态）
  • 使用函数或表达式的列

二、索引数据结构

2.1 B-Tree 索引

B-Tree（多路平衡查找树） 是一种自平衡的多路查找树。

特点：

  • 所有节点都存储数据
  • 叶子节点在同一层
  • 查找、插入、删除时间复杂度：O(log n)
  • 适合范围查询

2.2 B+Tree 索引

B+Tree 是 B-Tree 的变种，是 MySQL InnoDB 的默认索引结构。

B+Tree vs B-Tree：

对比项 B-Tree B+Tree
数据存储 所有节点 只存在叶子节点
非叶子节点 存数据和索引 只存索引
叶子节点 无链表 有双向链表
范围查询 需要中序遍历 直接遍历链表
查询稳定性 不稳定 稳定

为什么 InnoDB 选择 B+Tree：

  1. 非叶子节点不存数据，单页能存更多索引
  2. 树的高度更低，IO 次数更少
  3. 叶子节点有链表，范围查询效率高
  4. 全表扫描只需遍历叶子节点

2.3 Hash 索引

Hash 索引 基于哈希表实现，只适用于精确匹配。

特点：

  • 等值查询极快（O(1)）
  • 不支持范围查询
  • 不支持排序
  • 不支持模糊查询
  • InnoDB 的自适应哈希索引

2.4 数据结构对比

特性 B-Tree B+Tree Hash
等值查询 O(log n) O(log n) O(1)
范围查询 支持 支持（优秀） 不支持
排序 支持 支持 不支持
模糊查询 支持 支持 不支持
空间效率 一般 高 一般

3.2 按物理存储分类

聚簇索引（Clustered Index）：

  • 数据和索引存储在一起
  • 叶子节点存储完整的行数据
  • 一个表只能有一个聚簇索引
  • InnoDB 的主键索引就是聚簇索引

非聚簇索引（Secondary Index）：

  • 索引和数据分开存储
  • 叶子节点存储主键值
  • 一个表可以有多个非聚簇索引
  • InnoDB 的非主键索引都是非聚簇索引

  1. 为 WHERE 子句的列创建索引
  2. 为 JOIN 连接列创建索引
  3. 为 ORDER BY 和 GROUP BY 列创建索引

七、小结

核心知识点

  1. 索引数据结构：B+Tree 是 InnoDB 的默认结构
  2. 聚簇索引：数据和索引在一起，一个表只能有一个
  3. 非聚簇索引：索引和数据分开，可以有多个
  4. 索引分类：普通、唯一、主键、全文索引
  5. 性能分析：EXPLAIN 是最重要的分析工具

EXPLAIN 速查

字段 关注点
type ALL 最差，const 最好
key 是否使用了索引
rows 扫描行数，越少越好
Extra 避免 Using temporary、Using filesort
