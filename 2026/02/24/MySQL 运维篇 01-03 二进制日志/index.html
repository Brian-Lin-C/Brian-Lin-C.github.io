MySQL 运维篇 01-03 二进制日志

二进制日志（Binary Log，简称 binlog）是 MySQL 最重要的日志之一，记录了所有修改数据库结构或数据的操作。它是数据恢复和主从复制的基础。

一、二进制日志介绍

1.1 什么是二进制日志？

二进制日志是 MySQL 服务器层记录的日志，包含以下信息：

1.2 binlog 的作用

作用 说明
数据恢复 误删数据时，通过 binlog 恢复
主从复制 从库读取主库 binlog 并重放
审计追踪 记录所有数据变更历史
增量备份 配合全量备份实现增量恢复

1.3 binlog 格式

MySQL 支持三种 binlog 格式：

格式 说明 优点 缺点
STATEMENT 记录原始 SQL 日志量小 某些函数无法复制
ROW 记录行变更 数据精确 日志量大
MIXED 混合模式 兼顾两者 复杂

2.2 配置参数详解

log_bin：

  • 指定 binlog 文件的基础路径
  • 实际文件名会自动添加序号，如 mysql-bin.000001

expire_logs_days：

  • 自动删除过期的 binlog
  • MySQL 8.0+ 推荐使用 binlog_expire_logs_seconds

max_binlog_size：

  • 单个 binlog 文件的最大大小
  • 超过后自动切换到新文件

binlog_format：

  • STATEMENT：记录 SQL 语句
  • ROW：记录行数据变更
  • MIXED：默认 ROW，特殊情况下用 STATEMENT

sync_binlog：

  • 0：不同步，性能最好，风险最大
  • 1：每次事务都同步（默认，最安全）
  • N：每 N 次事务同步一次

四、binlog 格式详解

4.1 STATEMENT 格式

特点：记录原始 SQL 语句

优点：

  • 日志量小
  • 便于阅读和分析

缺点：

  • 某些函数无法正确复制（如 NOW()、UUID()）
  • 依赖确定性的 SQL

4.2 ROW 格式

特点：记录每行数据的变更

优点：

  • 精确记录数据变更
  • 不受函数和存储过程影响
  • 可以记录 DDL 以外的变更

缺点：

  • 日志量大
  • 不易直接阅读

4.3 MIXED 格式

特点：默认使用 ROW，特殊情况下用 STATEMENT

4.4 格式选择建议

场景 推荐格式
主从复制 ROW
数据恢复 ROW
日志量敏感 STATEMENT
默认推荐 ROW
